## 私密报告漏洞

使用 GitHub 的
[私密漏洞报告](https://github.com/4x25/agnes-compatible-gateway/security/advisories/new)，
不要提交公开 Issue。如果私密报告不可用，请通过仓库所有者的 GitHub 主页联系，
不要包含利用细节，只请求建立私密沟通渠道。

只提供安全复现所需的信息：

- 受影响版本、Git SHA、部署类型和接口；
- 影响及前置条件；
- 使用假凭据和非私人媒体的最小复现；
- 必要时提供确认不含秘密的 `X-Request-ID`；
- 已知的缓解建议。

不得发送真实 Agnes Key、Authorization、私人提示词、Base64 媒体、完整生成 媒体
URL、账户数据或无关个人信息。报告前先撤销任何可能泄露的 Key。

维护者目标是在 7 天内确认收到，并在 14 天内提供状态更新。修复时间取决于
严重程度，以及问题属于本项目、Agnes、Deno 还是其他依赖。公开细节前请留出
协调披露时间。

## 安全模型

- **BYOK：**生产运行时代码按请求接收调用方 Bearer Key；不读取
  `AGNES_API_KEY_ONLY_FOR_TEST`，也不存储 Key。
- **可信上游：**运营者控制 `AGNES_BASE_URL`。将其设为不可信主机，意味着
  按设计把调用方凭据交给该主机。
- **请求头隔离：**Authorization 只发给配置的 Agnes API 源，不发给最终媒体 存储
  URL。
- **不存储用户数据：**提示词和上传媒体只存在于请求处理期间；multipart 文件
  在内存转换，不使用数据库、缓存或持久队列。
- **浏览器测试台：**API Key 只保留在组件内存，不能进入 LocalStorage、URL
  参数、渲染 HTML、复制示例或遥测。
- **日志与错误：**不得包含凭据、请求体、Base64 媒体、完整上游媒体 URL 或
  内部堆栈。
- **资源边界：**上传、文件数量和聚合响应限制可减少意外内存耗尽，但公开部署
  仍需依赖平台级滥用防护。

## 范围内示例

- 凭据泄露到日志、错误、HTML、存储或错误来源；
- `/v1/*` 接口认证绕过；
- 暴露调用方凭据的 SSRF 或不安全重定向；
- 允许非预期浏览器凭据的 CORS 行为；
- 导致实质性拒绝服务的请求解析或流缺陷；
- 使文档部署方式存在实质风险的依赖或容器配置。

Agnes 账户失陷、模型输出质量/安全、Agnes 平台故障和托管平台限制通常属于
上游问题，但维护者会协助转交负责任报告的问题。
