## Browser and API coverage

The external Chromium run intentionally did not submit playground generation
requests because a remotely managed gateway cannot be redirected to the local
fake Agnes server. Its UI, accessibility, responsiveness, and credential-memory
checks all passed against the real deployment. The separately gated deployment
probe then exercised every platform-sensitive public workflow directly through
that same origin with a disposable caller key.

Image editing completed across the Deno Deploy request lifecycle, confirming
that the platform sustained the multipart upload and long upstream response.
Video creation, repeated stateless retrieval, terminal media resolution, and the
public byte-range proxy also completed. Generation requests were each made once;
only idempotent video reads were repeated.

## Release impact

The earlier
[release-candidate workflow](https://github.com/4x25/agnes-compatible-gateway/actions/runs/29655101517)
published an anonymously pullable OCI index with `linux/amd64` and
`linux/arm64`. Each architecture has its own SPDX and SLSA attestation manifest,
uses the unprivileged `deno` user, and embeds its immutable source SHA as
`DENO_DEPLOYMENT_ID`. The workflow inspected the remote index and successfully
started the published digest with a read-only filesystem and all capabilities
dropped.

Together, the real Deno Deploy run and GHCR candidate satisfy M5's deployment
and container acceptance gates. M5 remains in progress in the README until the
versioned source is tagged, the final multi-architecture image is verified, and
the public GitHub `v0.1.0` release exists.
